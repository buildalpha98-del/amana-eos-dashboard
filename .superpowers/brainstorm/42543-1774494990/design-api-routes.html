Design Section 4: API Routes & File Processing

New endpoints, indexing logic, and integration with existing document system

New API Routes

POST /api/knowledge/ask
Purpose: Main Q&A endpoint for the /knowledge page
Auth: withApiAuth — all roles
Body: { question: string } (Zod validated)
Flow: search chunks → build prompt → stream Claude response via SSE
Response: SSE stream with text deltas + final source metadata
Rate limit: 30 req/min (AI calls are expensive)
POST /api/knowledge/index
Purpose: Index a single document (called after upload)
Auth: withApiAuth — admin/owner/head_office
Body: { documentId: string }
Flow: download file from Vercel Blob → extract text → chunk → upsert DocumentChunks → update tsvector → set indexed=true
Response: { chunksCreated: number, tokenCount: number }
POST /api/knowledge/reindex
Purpose: Reindex all documents (admin utility)
Auth: withApiAuth — owner only
Body: none
Flow: fetch all non-deleted documents → delete existing chunks → re-extract and chunk all → update indexed flags
Response: { documentsProcessed: number, totalChunks: number, errors: string[] }
GET /api/knowledge/status
Purpose: Index status for the UI badge
Auth: withApiAuth — all roles
Response: { totalDocuments, indexedDocuments, totalChunks, lastIndexedAt, errors: { documentId, title, error }[] }

File Processing Library

src/lib/document-indexer.ts
// Core functions: extractText(fileUrl, mimeType) → PDF: pdf-parse (new dep) → DOCX: mammoth (already installed) → TXT/MD/CSV: fetch + read as text → Other: skip, set indexError chunkText(text, options?) → Split on headings (# / ## / bold lines) → Target ~500 tokens per chunk → 50-token overlap between chunks → Track heading context per chunk → Return: { content, heading, chunkIndex, pageNumber?, tokenCount }[] indexDocument(documentId) → Fetch Document from DB → Download file from Vercel Blob → extractText() → chunkText() → Delete old chunks (if re-indexing) → Insert new DocumentChunks in transaction → Raw SQL to set tsvector: UPDATE "DocumentChunk" SET "searchVector" = to_tsvector('english', content) WHERE "documentId" = $1 → Set document.indexed = true searchChunks(query, limit = 8) → Raw SQL with ts_rank: SELECT dc.*, d.title, d."fileName", ts_rank(dc."searchVector", plainto_tsquery('english', $1)) AS rank FROM "DocumentChunk" dc JOIN "Document" d ON d.id = dc."documentId" WHERE dc."searchVector" @@ plainto_tsquery('english', $1) AND d.deleted = false ORDER BY rank DESC LIMIT $2 → Return chunks grouped by document

Integration with Existing Upload Flow

Single Upload (POST /api/documents)

After creating the Document record, fire-and-forget call to indexDocument(id). The upload returns immediately — indexing happens async.

Bulk Upload (POST /api/documents/bulk)

After bulk creation, loop through created docs and call indexDocument(id) for each. Still fire-and-forget — bulk upload response doesn't wait.

Document Update/Delete

On update: re-index if fileUrl changed. On soft-delete: chunks remain but excluded from search (WHERE d.deleted = false). No orphan cleanup needed.

New Dependencies

pdf-parse

Extract text from PDFs. Lightweight, zero native deps, 500k weekly downloads.

mammoth

Already installed. Extract text from DOCX files.

That's it — one new package. Everything else uses Prisma raw SQL for tsvector operations.

Token Budget

Per question cost estimate:

  • System prompt: ~300 tokens
  • 8 retrieved chunks × ~500 tokens = ~4,000 tokens input
  • Response: ~500 tokens output
  • Total: ~4,800 tokens/question ≈ $0.02 on Haiku, $0.08 on Sonnet
  • Model: claude-haiku-4-5-20251001 (fast + cheap for retrieval Q&A)
