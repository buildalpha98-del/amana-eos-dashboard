Knowledge Agent — Architecture Approach

How should we store and search document content? ~100 documents (policies, SOPs, procedures, guides)

A

Claude Context Stuffing (Recommended)

Extract text from uploaded docs, store as chunks in Prisma. At query time, do keyword/fuzzy search to find relevant chunks, stuff them into Claude's context window as grounding material.

Pros

  • No new infrastructure — Prisma + Anthropic SDK only
  • ~100 docs = ~500 chunks easily searchable with PostgreSQL full-text search
  • Zero additional cost (no embedding API calls)
  • Simple to debug — you can see exactly what chunks were retrieved
  • Ships in 1-2 days

Cons

  • Keyword search can miss semantic matches ("sick child" won't match "illness management")
  • May need to tune chunk sizes for best results
  • Won't scale well past ~500 documents
B

Vector Embeddings (RAG)

Generate vector embeddings for each chunk using Voyage/OpenAI, store in pgvector extension. At query time, embed the question and do cosine similarity search for semantically relevant chunks.

Pros

  • Semantic search — "sick kid pickup" finds "Illness Management Policy"
  • Industry-standard RAG pattern
  • Scales to thousands of documents

Cons

  • Needs pgvector extension on Railway PostgreSQL
  • Adds embedding API dependency (Voyage ~$0.01/100 docs)
  • More complex to debug and maintain
  • Extra 2-3 days of work for marginal benefit at 100 docs
C

Hybrid: Full-Text Now, Vectors Later

Start with PostgreSQL full-text search (tsvector). Design the chunking/storage layer so vector embeddings can be added later as a drop-in upgrade when the doc count grows.

Pros

  • Ships fast like option A, with a clear upgrade path
  • PostgreSQL tsvector is more powerful than basic keyword search (stemming, ranking, phrases)
  • DocumentChunk table designed to accept an embedding column later
  • Best of both worlds — speed now, sophistication later

Cons

  • Slightly more upfront schema design than pure option A
  • Still no semantic search until vectors are added

My Recommendation: Option C (Hybrid)

At 100 documents, PostgreSQL full-text search with tsvector will handle 95% of queries well. It supports stemming ("managing" matches "management"), phrase search, and relevance ranking. The schema is designed so adding pgvector later is just adding a column + a search function swap — no rewrite needed. Ships in 2 days, upgradeable in 1 day later.
